## 1. プロジェクト概要

### 1.1 背景
中小企業やフリーランス事業者の多くは、業務マニュアル・議事録・仕様書・社内規程などのドキュメントが複数のフォルダやツールに散逸し、「どこに何が書いてあるか分からない」「新人が毎回同じ質問を繰り返す」といった課題を抱えている。汎用ChatGPTでは社内固有情報に回答できず、機密情報の学習利用リスクも存在する。

RAG（Retrieval-Augmented Generation）を用いることで、自社データを安全に検索対象としつつ、LLMの自然な回答生成能力を組み合わせた「社内専用AIアシスタント」を実現できる。

### 1.2 目的
- 社内ドキュメントの検索・参照時間を削減し、業務生産性を向上させる
- ハルシネーション（誤情報生成）を抑制し、根拠付きの回答を提供する
- 機密情報を外部に流出させないセキュアな運用を担保する
- ポートフォリオとして、リベシティワークス等での案件受注につなげる

### 1.3 ターゲットユーザー
| 区分 | 想定ユーザー | 主な利用シーン |
|------|-------------|---------------|
| 一次ターゲット | 従業員10〜50名規模の中小企業 | マニュアル・規程の検索、新人教育 |
| 二次ターゲット | 士業・コンサル・個人事業主 | 過去の提案書・ナレッジ再利用 |
| 三次ターゲット | リベシティ内の物販・EC事業者 | FAQ自動応答、業務マニュアル参照 |

### 1.4 解決する課題（ペインポイント）
1. 情報の鮮度問題：LLM単体では最新の社内情報に対応できない
2. ハルシネーション：根拠のない回答が業務判断を誤らせる
3. 情報の散逸：PDF/Word/Excel/Notionなど複数形式に跨る情報を横断検索できない
4. セキュリティ懸念：機密情報を外部AIサービスに投入するリスク

## 2. スコープ定義

### 2.1 MVP（今回の開発範囲）
- 複数形式ドキュメント（PDF / Word / Markdown / テキスト）のアップロードと取り込み
- ベクトルDBへのインデックス化（チャンク分割 + Embedding）
- チャット形式での自然言語質問
- 回答生成 + ソース引用表示（どのドキュメントの何ページか）
- シンプルなWeb UI（Streamlit または Next.js）
- ローカル / クラウド環境でのデプロイ

### 2.2 スコープ外（将来拡張）
- 詳細なユーザー権限管理（RBAC）
- Slack / Teams / Notion 等の外部連携（API統合）
- 多言語対応（日本語のみ）
- 音声入力 / 音声出力
- モバイルネイティブアプリ
- 回答精度のABテスト基盤

## 3. 機能要件

### 3.1 機能一覧

| ID | 機能名 | 優先度 | 概要 |
|----|-------|-------|------|
| F-01 | ドキュメントアップロード | 必須 | PDF/Word/MD/TXTをアップロードし、サーバに保存 |
| F-02 | ドキュメントのパースとチャンク分割 | 必須 | テキスト抽出し、意味単位でチャンクに分割 |
| F-03 | ベクトル化とインデックス登録 | 必須 | Embedding APIで埋め込み、ベクトルDBに保存 |
| F-04 | 自然言語による質問入力 | 必須 | チャット形式で質問を入力 |
| F-05 | ハイブリッド検索 | 必須 | ベクトル検索 + キーワード検索で関連チャンクを取得 |
| F-06 | 回答生成 | 必須 | 取得チャンクを文脈としてLLMが回答を生成 |
| F-07 | ソース引用表示 | 必須 | 回答の根拠となったドキュメント名・ページ番号を表示 |
| F-08 | チャット履歴の保持 | 推奨 | セッション内での会話の継続性を担保 |
| F-09 | フィードバック機能 | 推奨 | 回答に対する「役立った/的外れ」を記録 |
| F-10 | メタデータフィルタ | 推奨 | カテゴリ・作成日・部署などで検索範囲を絞る |
| F-11 | 管理画面（ドキュメント一覧・削除） | 任意 | アップロード済ファイルの管理 |

### 3.2 ユーザーフロー

```
[管理者] ドキュメントをアップロード
    ↓
[システム] パース → チャンク分割 → Embedding → ベクトルDB登録
    ↓
[利用者] チャット画面で質問を入力
    ↓
[システム] 質問をEmbedding化 → ハイブリッド検索で関連チャンク取得
    ↓
[システム] プロンプトに文脈を注入 → LLMが回答生成
    ↓
[利用者] 回答 + 引用元ドキュメントを確認 → フィードバック送信
```

## 4. 非機能要件

### 4.1 パフォーマンス
- 単一質問への回答時間：平均10秒以内（ドキュメント100ファイル規模）
- 同時利用者数：MVPでは10名程度を想定
- ドキュメントインデックス処理：100ページPDFで5分以内

### 4.2 セキュリティ
| 項目 | 要件 |
|------|------|
| API学習利用防止 | OpenAI等のAPIでオプトアウト設定を明示 |
| データ暗号化 | 通信経路のTLS化、ベクトルDB保管時の暗号化 |
| アクセス制御 | MVPでは単一テナント想定、将来的にRBAC拡張 |
| PII保護 | 個人識別情報（氏名・メール等）のマスキング機能を将来実装 |
| 監査ログ | 質問内容・回答・利用者IDを記録（90日保管） |
| 機密データ隔離 | 閉域VPC or ローカル環境での運用オプションを提供 |

### 4.3 可用性
- 稼働率：99%以上（月間ダウンタイム7時間以内）
- バックアップ：ベクトルDBおよびアップロードファイルを日次スナップショット

### 4.4 保守性
- コードカバレッジ：主要ロジックで70%以上
- ドキュメント：README、アーキテクチャ図、API仕様書を整備
- ログ出力：構造化ログ（JSON形式）でエラー追跡を容易化

## 6. データ要件

### 6.1 入力データ
| 種類 | 形式 | 制約 |
|------|------|------|
| ドキュメント | PDF / DOCX / MD / TXT | 1ファイル最大50MB、合計1GBまで（MVP） |
| 質問 | テキスト | 最大2,000文字 |

### 6.2 データ処理ルール
- **チャンク分割**：1チャンクあたり500〜1,000トークン、オーバーラップ100トークン
- **メタデータ**：ファイル名、ページ番号、作成日、カテゴリを各チャンクに付与
- **Embedding次元数**：1,536次元（text-embedding-3-small）

## 8. チケット駆動開発と手戻り最小化戦略

### 8.1 開発方針
本プロジェクトでは**チケット駆動開発（Ticket-Driven Development）**を採用する。実装・テスト時の手戻りが発生しても影響範囲を最小化するため、以下の原則を徹底する。

### 8.2 チケット粒度の原則
- **1チケット = 半日〜1日で完了する作業単位**に分割する
- **依存関係を明示**し、並行作業可能なチケットを区別する
- **完了条件（Definition of Done）をチケット起票時に明記**する
- **テスト観点を含めて起票**する（実装＋テストをセットで考える）
- **影響範囲を1モジュール内に閉じる**ように設計する（疎結合化）

### 8.3 チケット管理ツール
- GitHub Issues + GitHub Projects（カンバン形式）
- ラベル運用：`type:feature`, `type:bug`, `type:refactor`, `priority:high/mid/low`, `area:backend/frontend/infra`

### 8.4 手戻り最小化のための設計原則
| 原則 | 具体策 |
|------|-------|
| 疎結合 | RAG処理・UI・APIをモジュール分離、interfaceで依存性注入 |
| テストファースト | 必須機能（F-01〜F-07）はpytestで単体テスト先行記述 |
| 早期統合確認 | 第1週で「最小ループ（PDF1枚→質問→回答）」を通す |
| 仕様凍結ポイント | 各週末に仕様変更を凍結し、次週は実装のみに集中 |
| バージョン管理 | feature branchで開発、PR単位でレビュー（自己レビュー） |

## 9. UIチェックポイントとユーザビリティ確認

### 9.1 ユーザビリティ確認の目的
UIを作るタイミングで定期的にチェックポイントを設け、**「動くだけ」ではなく「使える」アプリ**を実現する。実装が進んでから根本的なUI変更を行うと手戻りが大きいため、早期かつ反復的に確認を行う。

### 9.2 チェックポイント一覧

| # | タイミング | 対象 | 確認観点 |
|---|----------|------|---------|
| ① | 第1週末（チケット #007） | 最小UI（PoC段階） | 全体フローが直感的か、致命的な使いにくさはないか |
| ② | 第3週前半（チケット #204） | チャット画面 | 質問入力・回答表示の操作感、引用元の視認性 |
| ③ | 第3週中盤（チケット #207） | 管理画面+全体 | アップロード手順の分かりやすさ、画面遷移 |
| ④ | 第3週末（チケット #212） | 本番環境 | デプロイ後の実環境での体感、レスポンス速度 |

### 9.3 各チェックポイントでの確認手順
1. **自己チェック**：開発者自身が「初見ユーザー」として一連の操作を実施
2. **第三者チェック（推奨）**：家族・友人・リベシティ仲間など非エンジニアに触ってもらう
3. **観察記録**：操作中に詰まった箇所・誤解した箇所・期待と異なる挙動をメモ
4. **改善チケット起票**：軽微な改善はその週内に対応、大きな変更は次週以降に切り出す
5. **判断基準**：致命的な使いにくさがあれば次工程に進まない（ストッパー機能）

### 9.4 ユーザビリティ評価項目
- **学習しやすさ**：初見で5分以内に主要機能を使えるか
- **効率性**：質問入力から回答取得まで何クリックか
- **エラー回復**：誤った操作からの復帰が容易か
- **満足度**：「もう一度使いたい」と思えるか
- **視認性**：重要情報（回答・引用元）が視覚的に目立つか

## 10. ポートフォリオ用デモ動画制作

### 10.1 目的
- リベシティワークスや採用担当者に**「短時間で技術力と価値を伝える」**手段
- 文字や静止画では伝わらない**動的な体験・操作感**を訴求
- ポートフォリオ全体の**信頼性・完成度の象徴**として機能

### 10.2 動画仕様

| 項目 | 内容 |
|------|------|
| 長さ | 3分以内（注意持続限界を考慮） |
| 解像度 | 1080p以上 |
| 音声 | ナレーション付き（または字幕付き） |
| 公開先 | YouTube（限定公開）、Loom、READMEに埋め込み |
| 収録ツール | Loom / OBS Studio / QuickTime |
| 編集ツール | DaVinci Resolve（無料）/ CapCut |

### 10.3 動画構成（シナリオ）

| 時間 | セクション | 内容 |

| 0:00–0:20 | オープニング | 課題提起：「社内ドキュメント、どこに何があるか分からない問題」 |

## 13. リスクと対策

| リスク | 影響度 | 対策 |
|--------|-------|------|
| 回答精度が低い（ハルシネーション多発） | 高 | チャンク分割戦略の最適化、プロンプト改善、ハイブリッド検索導入 |
| API利用料が想定超過 | 中 | GPT-4o-miniなど低コストモデルを優先、利用量アラート設定 |
| 開発スケジュール遅延 | 中 | チケット粒度を細かくし、優先度順に着手。Dify等のローコードツールで一部代替 |
| 実装後のUI変更で手戻り | 高 | UIチェックポイントを4回設置、早期に課題を検出 |
| セキュリティ懸念でクライアントが導入を躊躇 | 高 | オプトアウト設定・ローカル運用オプション・監査ログを明示 |
| RAGプロジェクト全体の成功率が低い（業界調査で約33%） | 高 | データ前処理・メタデータ設計・継続改善ループを要件に明記 |
| デモ動画制作に時間がかかる | 中 | シナリオ事前確定、収録は複数テイク短時間、編集は最低限に |
